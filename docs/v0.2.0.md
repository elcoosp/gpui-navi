## Vision Statement

> **Vision:** Navi Router becomes the Go‑to Rust routing library that seamlessly integrates with any UI framework (GPUI, Dioxus, egui, desktop/web), while maintaining its current high‑performance, file‑based developer experience.

## Elevator Pitch (Moore’s template)

For **Rust application developers** who need a **type‑safe, file‑based router** but are locked into a specific UI framework, **Navi Router** is a **freshly decoupled routing library** that provides a **pure, framework‑agnostic core** and **first‑class GPUI adapter**. Unlike the current monolithic implementation, our product **allows you to use the same routing logic with Dioxus, egui, or any custom Renderer, shortening integration time from weeks to hours.**

## Problem Statement & Business Context

The current `navi-router` crate is tightly coupled to GPUI. Every component, state management, and even loader execution is tied to `gpui::App` and `Window`. This prevents:

- **Reuse** of routing logic in non‑GPUI projects.
- **Ecosystem growth**: other UI framework users cannot adopt Navi Router.
- **Testing** of routing behavior without spinning up a full GPUI application.
- **Maintainability**: core routing logic mixed with UI‑specific concerns is harder to evolve and review.

Now that the router has proven itself in the GPUI example app, it is time to split it into a **standalone core** and a **thin adapter layer**, enabling broader adoption and future‑proofing the architecture.

## Target Users / Customers

**Primary:** Rust developers building applications with GPUI (current users) who need long‑term maintainability and cleaner internal APIs.
**Secondary:** Rust developers using other UI frameworks (Dioxus, egui, tauri‑native) who want a file‑based, type‑safe router.
**Explicit non‑target:** Developers who prefer hand‑written route tables without code generation; we will not build a “no‑codegen” alternative in this release.

## User Needs & Value Proposition

1. **Framework‑agnostic routing core** – define routes, match paths, manage history, and handle loaders without importing GPUI.
2. **Clean separation** – existing GPUI projects upgrade with minimal changes, while new adapters can be written for any framework.
3. **Preserve all existing features** – the code generator, macros, devtools, and all route features should work identically after the refactor.
4. **Better testability** – the core can be unit‑tested without a window.

**Differentiator:** Unlike other Rust routers that are framework‑specific, Navi Router will offer a pluggable architecture: one core, many adapters.

## Desired Outcomes & Success Metrics

| Business Objective | Key Results |
|--------------------|-------------|
| OBJ‑01: Deliver a pure, framework‑agnostic routing core | KR1: `navi-router-core` crate compiles with **zero** GPUI dependencies. KR2: Core passes a full suite of unit tests without any GPUI context. |
| OBJ‑02: Maintain full backward compatibility for GPUI users | KR1: Example‑app compiles and runs without changes to route files. KR2: All existing 20+ route files still register and work as before. |
| OBJ‑03: Enable integration with at least one other UI framework (Dioxus) as a proof‑of‑concept | KR1: Adapter crate `navi-dioxus` exists and passes a minimal “hello‑world” demo. KR2: Time to add a new framework adapter is under 2 person‑days. |
| OBJ‑04: Improve internal modularity & test coverage | KR1: Core crate has >80% code coverage. KR2: All pure logic (tree, history, blockers, validation) resides in core, no adapter duplication. |

## Strategic Constraints

- **Release deadline:** Must ship within the next 2‑month cycle.
- **No breaking changes to route file syntax or `define_route!` macro.**
- **Must remain a Cargo workspace** with existing crates; no mega‑refactor that forces a complete rewrite.
- **Rust stable only**; no nightly features.

## Goals & Non‑goals (Scope & Anti‑scope)

**Goals (what we commit to deliver):**
- [G‑1] Extract a `navi-router-core` crate containing `Location`, `History`, `RouteTree`, `Blocker`, `Validation`, `RouteNode`, `Segment`, etc.
- [G‑2] Refactor `navi-router` to become a GPUI adapter that wraps `RouterCore`.
- [G‑3] Introduce a clean `RouterCore` API with `NavigationEffect` enum, no direct `App` usage.
- [G‑4] Keep all builder macros and codegen working with the new structure.
- [G‑5] Update devtools to work with the adapter layer.
- [G‑6] Provide documentation on how to write a new adapter.

**Non‑goals (explicit exclusions):**
- [NG‑1] **We will not** build a full‑featured Dioxus adapter in this release; only a minimal proof‑of‑concept or integration test.
- [NG‑2] **We will not** change the route‑file discovery or codegen format.
- [NG‑3] **We will not** add new routing features (like route transitions, full‑async loading). Feature development is paused.
- [NG‑4] **We will not** introduce a different macro system (like a derive‑based route definition). `define_route!` remains the sole entry point.
- [NG‑5] **We will not** optimize performance beyond current levels; the goal is parity, not faster.
- [NG‑6] **We will not** remove the GPUI‑specific `RouterProvider`; it remains as the primary adapter.

## Operational Concept & High‑Level Scenarios

1. **Existing GPUI Application Upgrade**
   A developer updates `Cargo.toml` dependencies, and the code compiles unchanged. All routes, loaders, and devtools continue to work. Internally, `RouterState` is now a thin wrapper around `RouterCore` + `QueryClient`.

2. **Building a New GPUI Application**
   The workflow stays the same: define routes with files and `define_route!`, run the code generator, and use `RouterProvider` as before. No new APIs required.

3. **Using Core in a Non‑GPUI Context**
   A developer adds `navi-router-core` to a non‑GPUI project (e.g., a CLI tool testing route logic). They instantiate `RouterCore`, add routes manually, and call `navigate()`. Navigation effects are returned as an enum; the developer decides how to execute them (e.g., with `tokio::spawn`).

4. **Exploring a New Adapter (e.g., Dioxus)**
   A contributor studies the GPUI adapter source, sees how `NavigationEffect` is translated into framework‑specific actions (spawning, refreshing), and writes a similar adapter. The core remains unchanged.

## 1. Introduction & Scope

This SRS defines the functional and non‑functional requirements for the decoupling of the Navi Router library. The primary goal is to extract a pure `navi-router-core` crate while preserving all existing functionality for GPUI users. The scope includes the new core, the refactored GPUI adapter, the build tooling adjustments, and a proof‑of‑concept Dioxus adapter.

**Out of scope:** new routing features, UI framework changes, new validation crates.

## 2. System Context & Overview

**Product:** Navi Router, an open‑source, file‑based routing library for Rust.  
**Context after decoupling:**  
- `navi-router-core` – pure library, used by adapters.  
- `navi-router` – GPUI adapter, used by GPUI applications.  
- `navi-codegen` – code generator, generates adapter‑specific route tree registration.

## 3. Functional Capabilities & Behavior

### Feature: Core Crate Extraction

**REQ‑FUNC‑001 (Core zero‑GPUI dependency):** The `navi-router-core` crate shall compile with no dependency on any GPUI crate or any crate that requires a GUI runtime.  
**Acceptance:** Cargo metadata shows no `gpui` in dependency tree.

**REQ‑FUNC‑002 (Core public API):** The core shall expose: `Location`, `History`, `RouteTree`, `RouteNode`, `Block`, `Redirect`, `NotFound`, `RouterCore`.  
**Acceptance:** The listed types appear in `pub use` and are documented.

**REQ‑FUNC‑003 (RouterCore navigate returns effects):** `RouterCore::navigate()` shall take a `Location` and return a `Vec<NavigationEffect>` (or similar) describing the side‑effects required (spawn loader, redirect, notify), without performing any I/O or framework‑specific actions.  
**Acceptance:** Unit test: simulate a navigation to a route with a loader; verify the returned effects include a `SpawnLoader` effect with the expected key.

### Feature: GPUI Adapter Refactor

**REQ‑FUNC‑004 (Backward‑compatible RouterProvider):** The refactored `RouterProvider` shall accept the same constructor parameters and produce the same render tree as before.  
**Acceptance:** Existing example app compiles without changes to `main.rs`.

**REQ‑FUNC‑005 (Loader execution):** When a `NavigationEffect::SpawnLoader` is emitted, the GPUI adapter shall use `cx.spawn` and the existing `QueryClient` to execute the loader, exactly as before.  
**Acceptance:** Route with loader still shows loading → data flow in example app.

**REQ‑FUNC‑006 (Devtools integration):** The devtools overlay shall continue to display routing events and history. Internally, it may obtain events from the adapter (not directly from core).  
**Acceptance:** Navigate between routes and observe devtools timeline updates.

### Feature: Code Generation Compatibility

**REQ‑FUNC‑007 (Unchanged route file syntax):** Route files using `define_route!` shall require no modification.  
**Acceptance:** All existing route files compile as‑is.

**REQ‑FUNC‑008 (Generated route tree):** The code generator shall produce a `build_route_tree()` and `register_routes()` that target the adapter’s route types. The generated code must not reference core APIs directly (to keep adapter flexibility).  
**Acceptance:** Generated `route_tree.gen.rs` files compile with the refactored workspace.

### Feature: Proof‑of‑Concept Dioxus Adapter

**REQ‑FUNC‑009 (Minimal Dioxus adapter):** There shall exist a `navi-dioxus` crate that wraps `RouterCore` and provides a `DioxusRouter` component that renders an `Outlet` equivalent for Dioxus.  
**Acceptance:** The PoC app navigates between two simple routes without errors.

## 1. Context & Scope

This architecture document describes the refactoring of Navi Router from a monolithic GPUI‑coupled library into a set of modular crates: a pure core and a GPUI adapter. The primary driver is to make the router framework‑agnostic and improve maintainability.

**Scope:** `navi-router-core`, `navi-router` (adapter), `navi-codegen`, and the PoC Dioxus adapter.

## 2. Goals & Non‑goals (Design Level)

**Goals:**
- Remove all GPUI dependencies from the core routing logic.
- Define a clean `NavigationEffect`‑based contract between core and adapter.
- Keep the existing file‑based routing, macros, and devtools compatible.
- Provide a proof‑of‑concept for a second adapter (Dioxus).

**Non‑goals:**
- Do not change the route definition syntax or code generation output format.
- Do not add new runtime features (e.g., middleware, router guards) in this release.
- Do not break the current loader execution model; just relocate it behind an effect.

## 3. Architecturally Significant Requirements (ASRs)

| ID | ASR | Source |
|----|-----|--------|
| ASR‑001 | Core must compile without GPUI. | NFR‑COMPAT‑001, OBJ‑01 |
| ASR‑002 | Navigation must not cause side effects in the core; effects are returned for the adapter to interpret. | REQ‑FUNC‑003 |
| ASR‑003 | Route matching performance must not regress. | NFR‑PERF‑001 |
| ASR‑004 | All existing route files (20+) must load and function identically. | OBJ‑02 |
| ASR‑005 | Loader execution stays in the adapter (GPUI) and uses the same `rs-query` client. | REQ‑FUNC‑005 |

## 4. The Design

### 4.1 System Overview (C4 Context / Container)

**Before:**  
`navi-router` (contains all logic + GPUI integration) → `gpui`  
`navi-codegen` (generates code that directly uses `navi-router`)

**After:**  
```
navi-router-core (pure) ← navi-router (adapter, depends on gpui + core)
                        ← navi-dioxus (adapter, depends on dioxus + core)
navi-codegen (generates adapter‑specific registration code for each adapter)
```

### 4.2 Core Architecture – `RouterCore`

`RouterCore` is a state machine that owns:
- `History`
- `RouteTree`
- `current_match`
- `blockers`
- `pending_navigation`

Its `navigate(&mut self, loc: Location)` method returns a `Vec<NavigationEffect>`. Effects are:

```rust
enum NavigationEffect {
    SpawnLoader { route_id: String, params: HashMap<String, String> },
    Redirect { to: Location, replace: bool },
    NotFound { data: Option<serde_json::Value> },
    NotifyListeners,
}
```

The core does **not** have a `QueryClient` or any knowledge of `cx.spawn`. The adapter receives effects and executes them.

### 4.3 GPUI Adapter – `GpuiRouter`

`GpuiRouter` wraps `RouterCore` and a `QueryClient`. It implements `Global` as the existing `RouterState` did.  

When it calls `core.navigate()`, it receives effects. For `SpawnLoader`, it uses `cx.spawn` and the `QueryClient` to execute the loader, exactly as today. `NotifyListeners` triggers `cx.refresh_windows()`.

All existing `RouterState::update`, `Navigator`, etc., remain as convenience methods.

### 4.4 Code Generation

`navi-codegen` remains directory‑based. The generated `build_route_tree()` now constructs a `RouteTree` (core type) and then passes it to `adapter::RouteTree` (adapter wrapper). The generated `register_routes` calls the adapter’s registration.

No changes to `navi.config.json`.

### 4.5 Dioxus Adapter PoC

A minimal `navi-dioxus` crate:  
- Owns `RouterCore` in a `DioxusRouter` component.  
- Listens to `NavigationEffect`s and uses Dioxus’s `use_future` or `spawn` to perform loaders.  
- Renders an `Outlet` component that reads the current route’s component from a registry (similar to the existing `REGISTRY`).

## 5. Architecture Decision Records (ADRs)

### ADR‑001: Extract core with `NavigationEffect` pattern

**Context:** Need to decouple routing logic from GPUI.  
**Decision:** Move all pure logic into `navi-router-core`. Introduce `NavigationEffect` enum to communicate side‑effects to adapters.  
**Consequences:** Core becomes testable without GPUI; adapters must implement effect handling, but this is well‑contained.  
**Drivers:** ASR‑001, ASR‑002.

### ADR‑002: Keep `define_route!` macro unchanged

**Context:** The macro generates code that currently depends on GPUI types.  
**Decision:** The macro remains as is; the generated code will be linked against the adapter’s types. The macro’s expansion uses the adapter’s re‑exports of core types.  
**Consequences:** No user‑visible changes; macro stays framework‑specific (acceptable).  
**Drivers:** ASR‑004.

### ADR‑003: Devtools stay in GPUI adapter

**Context:** Devtools currently render with GPUI.  
**Decision:** Keep devtools in `navi-router` (adapter) and let them subscribe to adapter‑level events. Event logging can be pure but remains in the adapter for simplicity.  
**Drivers:** TBD‑02.

### ADR‑004: Dioxus adapter as separate crate, minimal scope

**Context:** Need to validate adapter pattern.  
**Decision:** Create `navi-dioxus` as a workspace member; it will contain only a single example app. Not intended for production use.  
**Drivers:** OBJ‑03.

*(Full ADR templates with alternatives would be created per decision.)*

The new core’s public API stabilizes on the following exports (subject to minor adjustments during implementation):

```rust
pub mod location;

pub mod redirect;

- **Testing:** Core unit tests run with `cargo test`; no GPUI context needed.  
- **Observability:** Event emission from core can be captured by adapters for logging/devtools.

## 8. Alternatives Considered

1. **Keep monolith but add feature flags:** Would not achieve zero‑framework dependency. Rejected because core cannot be used independently.

```rust
impl RouterState {
    pub fn navigate(&mut self, loc: Location, options: NavigateOptions, cx: &mut App) {
        let effects = self.core.navigate(loc, options);
        
        for effect in effects {
            match effect {
                NavigationEffect::SpawnLoader { route_id, params } => {
                    if let Some(factory) = self.loader_factories.get(&route_id) {
                        let query = factory(&params);
                        let key = query.key.clone();
                        let client = self.query_client.clone();
                        let fetch_fn = query.fetch_fn.clone();
                        let options = query.options.clone();
                        let window_handle = self.window_handle;
                        
                        cx.spawn(|cx: &mut gpui::AsyncApp| async move {
                            match (fetch_fn)().await {
                                Ok(outcome) => {
                                    client.set_query_data(&key, outcome, options);
                                    let _ = cx.update(|cx| {
                                        cx.refresh_windows();
                                        let _ = window_handle.update(cx, |_, window, _| window.refresh());
                                    });
                                }
                                Err(e) => {
                                    log::error!("Loader error: {:?}", e);
                                }
                            }
                        }).detach();
                    }
                }
                NavigationEffect::Redirect { to, replace } => {
                    let nav = crate::Navigator::new(self.window_handle);
                    if replace {
                        nav.replace(to, cx);
                    } else {
                        nav.push(to, cx);
                    }
                }
                NavigationEffect::NotFound { data } => {
                    self.not_found_data = data;
                    let nav = crate::Navigator::new(self.window_handle);
                    nav.push("/404", cx);
                }
                NavigationEffect::NotifyListeners => {
                    cx.refresh_windows();
                    let _ = self.window_handle.update(cx, |_, window, _| window.refresh());
                }
            }
        }
    }
}
```

# Navi Router Architecture

## Core

`navi-router-core` contains all pure routing logic with zero UI framework dependencies:
- `RouterCore` - state machine; navigate returns `NavigationEffect` enum
- `RouteTree` - radix trie for O(log n) path matching
- `History` - browser-style history stack

## Adapters

Adapters wrap `RouterCore` and interpret `NavigationEffect`s for a specific UI framework:
- `navi-router` - GPUI adapter (uses `cx.spawn`, `QueryClient`)
- Future: Dioxus, egui, etc.

## Writing a New Adapter

1. Instantiate `RouterCore` with a `RouteTree`
2. On navigation, call `core.navigate(loc)` and handle effects
3. For `SpawnLoader`, use your framework's async runtime
4. For `NotifyListeners`, refresh your UI
```
