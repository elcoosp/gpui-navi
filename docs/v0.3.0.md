## 1. Behavioral Specifications (Specification by Example)

### Feature: Core Router Independence

**Scenario:** Core match static path without GPUI context  
Given a `RouterCore` with routes `/` (home) and `/about`  
When `core.navigate(Location::new("/about"), default_options)` is called  
Then the returned effects contain only `NotifyListeners`  
And `core.current_match` contains the “about” node.

**Scenario:** Core match dynamic path  
Given a `RouterCore` with routes `/users/$id`  
When `core.navigate(Location::new("/users/42"), default_options)` is called  
Then the returned effects include `SpawnLoader` if the route has a loader, or `NotifyListeners` if not  
And the current match’s params contain `{"id": "42"}`.

**Scenario:** Core returns NotFound for unknown path  
Given an empty `RouteTree`  
When `core.navigate(Location::new("/missing"), default_options)` is called  
Then the returned effects contain only `NotFound` (data may be None)  
And `current_match` remains unchanged.

**Scenario:** Core preserves history after navigation  
Given a core with routes `/`, `/about` starting from `/`  
When `core.navigate(Location::new("/about"), default_options)` is called  
Then `core.current_location().pathname` equals `/about`  
And `core.history().can_go_back()` is true.

### Feature: GPUI Adapter Backward Compatibility

**Scenario:** Link click renders correct page  
Given the example app started with the refactored router  
When the user clicks the “About” link  
Then the window title (or rendered text) includes “About” content  
And the route tree is still generated from the same route files.

**Scenario:** Loader shows loading then data  
Given the `/awaited` route with a 2‑second loader  
When the user navigates to `/awaited`  
Then initially “Loading…” is displayed  
And after the loader completes, “Loaded data!” is shown.

**Scenario:** Navigation with search params  
Given the user is on `/users` with `?sort=asc`  
When the user clicks the “↓ Desc” button  
Then the browser URL changes to `/users?sort=desc`  
And the list of users is re‑ordered accordingly.

**Scenario:** Blocker prevents navigation  
Given the settings page has a blocker active  
When the user tries to navigate away by clicking “Home”  
Then navigation is blocked (the URL does not change)  
And the settings page stays in view.

**Scenario:** before_load redirects unauthenticated user  
Given the admin route is configured with `before_load` that redirects to `/login`  
When an unauthenticated user navigates to `/admin`  
Then the router redirects to `/login`  
And the admin layout is never rendered.

**Scenario:** Devtools timeline shows navigation events  
Given the example app with devtools visible  
When the user navigates from `/` to `/about`  
Then the devtools timeline shows a `NAV` event with `/ → /about`  
And after load, it shows `DONE` or `OK`.

### Feature: Code Generation Compatibility

**Scenario:** Generated `route_tree.gen.rs` compiles unchanged  
Given the existing route files in `example-app/src/routes/`  
When `cargo build` is run  
Then `route_tree.gen.rs` is regenerated without errors  
And it uses `navi_router::RouteTree` as the route tree type.

**Scenario:** All route macros (`define_route!`) work the same  
Given all route files using `define_route!`  
When the example app compiles  
Then all route components are registered  
And the generated `build_route_tree()` produces the same number of routes as before.

### Edge Cases & Unwanted Behavior

**Scenario:** Loader error is handled gracefully  
Given a route whose loader returns `Err(...)`  
When the user navigates to that route  
Then the UI does not crash; an error is logged, and the fallback UI (e.g., “Loading…” remains) is displayed.

**Scenario:** Concurrent navigation requests are handled  
Given the user rapidly clicks two different links  
Then only the last navigation is resolved (no race conditions)  
And the route tree match is consistent.

**Scenario:** Missing route file does not break codegen  
Given the `routes_directory` has a file with no `define_route!`  
When the code generator runs  
Then it treats it as a stub and does not error.

**Scenario:** Empty `routes_directory` yields a valid `RouteTree`  
Given no route files exist  
When `build_route_tree()` is called  
Then an empty `RouteTree` is returned and no panic.

## 2. Test Strategy & Plan

### Approach

We follow a layered test pyramid with emphasis on core unit tests and integration tests for the adapter.

- **Unit tests** – every pure function in `navi-router-core` (history, trie, blocker, validation, core navigation).
- **Integration tests** – adapter + core interaction via a headless GPUI environment (if feasible) or via the example app’s compile‑time checks.
- **Codegen output validation** – snapshot or assertion tests that verify the generated `route_tree.gen.rs` matches expectations.
- **Manual exploratory** – the example app is exercised for critical flows (navigation, blockers, before_load, devtools) before release.
- **Performance regression** – benchmark route matching and core navigation against baseline from pre‑refactor build.

### Tools

- `cargo test` for all automated tests.
- Property‑based testing (`proptest`) optionally for history stack operations.
- `cargo bench` or custom benchmark binary for performance.
- Manual inspection of example app.

### Risk‑Based Prioritization

High risk areas (loader lifecycle, blocker interaction, before_load) get thorough integration coverage. Low risk (codegen formatting) is verified by snapshot/compilation.

## 4. NFR Verification Plans

### Performance (NFR‑PERF‑001)
- **Verification method:** Benchmark test (Analysis + Test)
- **Tool:** A custom binary that inserts 10,000 routes with varying patterns and matches 100 random paths, measuring average time.
- **Baseline:** Pre‑refactor build benchmark result stored in a file.
- **Acceptance:** Post‑refactor average match time must not exceed 110% of baseline.

### Maintainability (NFR‑MAIN‑001)
- **Verification method:** Tool‑based measurement (Analysis)
- **Tool:** `cargo tarpaulin` for code coverage
- **Acceptance:** `navi-router-core` line coverage ≥ 80%.

### Compatibility (NFR‑COMPAT‑001)
- **Verification method:** Inspection + Build
- **Acceptance:** The core crate compiles with `edition = 2024` and the minimum Rust stable version used in CI. The example app compiles and runs without any route file modifications.
